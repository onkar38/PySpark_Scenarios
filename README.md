# PySpark Scenarios — Silent Failures in Production

> 6 real-world PySpark bugs that don't crash your job — they silently produce wrong results.  
> Every scenario includes a **Before** (broken) and **After** (fixed) with execution time proof.

---

## What This Project Does

Spark never throws an error when your logic is wrong. It exits with code 0 — and the bug shows up 3 days later in a finance review.

This repo demonstrates 6 production scenarios on **10 million rows** where Spark silently:
- Produces wrong revenue numbers
- Misses fraud signals
- Wastes network bandwidth
- NULLs 20% of your data
- Makes a job slower *because* you cached it
- Triggers a full shuffle on already-aggregated data

Each notebook has a **Before** and **After** with actual execution times.

## Scenarios

### Scenario 1 — Data Skew · Salting
**File:** `scenario_1.ipynb`  
**Execution time:** 36.2s → 11.2s (**3x faster**)

**Problem:** 10 merchants hold 70% of 10M rows. A plain `groupBy("merchant_id")` assigns 700,000 rows to a few partitions while others get ~600. The hot partitions become a bottleneck — all other executors sit idle waiting.

**Fix:** Detect hot keys → append a random salt suffix → group on salted key → de-salt and re-aggregate.

```python
# Detect hot keys
hot_keys = df.groupBy("merchant_id").agg(count("*").alias("cnt")).filter(col("cnt") > 100000)

# Salt hot rows
salt_df = hot_rows.withColumn(
    "salt_key",
    concat(col("merchant_id"), lit("_"), (rand() * 10).cast("int"))
)
```

### Scenario 2 — Fraud Detection · Window Functions
**File:** `scenario_2.ipynb`  
**Problem:** Fraud missed for 4 days.

**Problem:** A customer makes 3 transactions worth ₹85,000 in 8 minutes across 3 merchants. `GROUP BY` counts total transactions per day — it shows nothing abnormal. The fraud team spots it 4 days later manually.

**Root cause:** `GROUP BY` cannot see time gaps between transactions. It only aggregates totals.

**Fix:** Window function with a rolling 10-minute time range detects burst patterns.

```python
w_time = Window.partitionBy("customer_id").orderBy("ts").rangeBetween(-600, 0)
w_cust = Window.partitionBy("customer_id")

df = df.withColumn("avg_amt", avg("amount").over(w_cust))
df = df.withColumn("high_flag", when(col("amount") > col("avg_amt") * 3, 1).otherwise(0))
df = df.withColumn("high_txn_count", sum("high_flag").over(w_time))

fraud = df.filter(col("high_txn_count") >= 3)
```

### Scenario 3 — Unnecessary Shuffle · Broadcast Join
**File:** `scenario_3.ipynb`

**Problem:** A 500-row, ~20KB `merchant_lookup` table is shuffle-joined against 10M transaction rows. Every executor ships data across the network for a table smaller than most log files.

**Fix:** `broadcast(merchant)` sends the small table to every executor once. Zero network movement on the large side.

```python
# Before — default shuffle join (network overhead on 10M rows)
transaction.join(merchant, "merchant_id", "left")

# After — broadcast join (20KB sent once to each executor)
transaction.join(broadcast(merchant), "merchant_id", "left")
```

### Scenario 4 — Schema Mismatch · mergeSchema
**File:** `scenario_4.ipynb`  
**Silent impact:** 20% revenue drop, no error thrown.

**Problem:** Chunks 0–7 use column `amount`. Chunks 8–9 (2M rows from the new payments team) renamed it to `transaction_amount`. Without `mergeSchema`:
- **Mode 1:** Spark reads old files first — NULLs 2M rows silently. 20% of revenue = 0.
- **Mode 2:** Spark reads new files first — drops `amount` entirely. 8M rows contribute nothing.

Finance team notices 3 days later.

**Fix:** `mergeSchema=True` + `coalesce` to unify both column names.

```python
transaction = spark.read.option("mergeSchema", "true").parquet(path)

# Unify both column names into one
transaction = transaction.withColumn(
    "amount",
    coalesce(col("amount"), col("transaction_amount"))
)
transaction.groupBy("customer_id").agg(sum("amount").alias("total_revenue")).show()
```

### Scenario 5 — Wrong Caching · Cache the Right Subset
**File:** `scenario_5.ipynb`

**Problem:** Caching the full 10M-row table (3–5GB) on a 1.5GB executor memory pool causes silent disk spill. Disk reads are 100x slower than memory. The `.cache()` call made the job *slower*, not faster.

**Fix:** Filter to only the rows you need (Gold tier merchants = ~400–600MB), then cache that subset. It fits in memory. All 3 downstream reuses are instant.

```python
# Wrong — full table spills to disk
df_raw.cache()

# Correct — cache only the subset that fits in memory
filter_df = df_raw.filter(col("tier") == "Gold")
filter_df.cache()
filter_df.count()  # materialise once

# All 3 reuses are now instant memory reads
filter_df.groupBy("merchant_id").agg(count("*")).show()
filter_df.groupBy("merchant_name").agg(count("*")).show()
filter_df.groupBy("category").agg(count("*")).show()
```

### Scenario 6 — repartition(4) vs coalesce(4)
**File:** `scenario_6.ipynb`  
**Execution time:** 18.8s → 1.6s (**10x faster**)

**Problem:** After a `groupBy` aggregation, using `repartition(4)` to write 4 compact parquet files triggers a **full second shuffle** on already-aggregated data. An Exchange node appears in the physical plan. Network bytes move again for no reason.

**Fix:** `coalesce(4)` merges nearby partitions locally — no network movement, no Exchange node in the plan. Same output files, 10x faster.

```python
# Wrong — full shuffle on aggregated data
df_merchant_revenue.repartition(4).write.parquet("data/gold/merchant_revenue/")

# Correct — local merge, zero network movement
df_merchant_revenue.coalesce(4).write.parquet("data/gold/merchant_revenue/")
```
